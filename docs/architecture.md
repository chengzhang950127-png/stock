# 量化交易系统架构决策文档

**文档定位**：本项目的产品愿景、架构原则、核心机制的权威说明。任何代码实现必须符合本文档的约束。

**修订规则**：架构调整必须先更新本文档再调整代码。所有 WP 评审都基于本文档校验一致性。

---

## 1. 产品愿景

构建一个**LLM 增强的多策略量化交易系统**，重点支持美股和港股，定位是"个人量化策略孵化器"。核心特征：

- 多套理论独立的策略并行运行（Champion-Challenger 框架）
- 用户可定制策略（基于内置策略的加权组合）
- 投资助手主动识别市场状态、推荐策略权重和资产配置
- 公开访问（未登录可见所有策略表现和建议）+ 私有托管（登录后可托管真实账户）
- 代码主导的稳定架构 + 必要环节的 AI 增强

产品的核心价值在于**让用户长期跟踪多套策略的真实表现**，建立对策略的诚实判断，并在合适时候切换到合适策略。

## 2. 六层架构

系统分为六个职责单一的层级，自上而下数据流单向清晰：

### 2.1 数据基础层

**职责**：从外部数据源接入、归一化、入库、缓存。

**包含**：
- 美股行情数据适配器（YFinance / Polygon）
- 港股行情数据适配器（AkShare / Tushare Pro，含南向资金）
- 宏观经济数据适配器（FRED 等）
- 新闻和事件数据适配器（财经新闻流 + FOMC/财报日历）
- 数据存储（PostgreSQL 生产，SQLite 开发）
- 缓存层

**性质**：纯代码，无 AI 调用。

### 2.2 策略引擎层

**职责**：运行多套策略，每天产出选股、信号、买卖价位。

**包含**：
- 4 套内置永久策略：价值反转、趋势动量、事件驱动、指数跟踪（GTAA）
- 0-2 个用户自定义策略（V0.5 起支持，加权组合形式）
- 策略基类 `StrategyBase` + 因子库 + 信号生成工具
- 策略库（活跃池 + 历史归档）
- 策略生命周期管理器（活跃 ↔ 归档 ↔ 删除 + 回捞流程）

**性质**：代码主导。事件驱动策略调用 AI 服务解析新闻（**第一个允许的 AI 调用点**）。其他策略纯代码。

### 2.3 回测与持仓管理层

**职责**：用历史数据驱动策略 + 维护影子账户 + 计算 P&L。

**包含**：
- 向量化回测引擎
- 账户状态机（影子账户 + 实施账户）
- 持仓管理（开仓/平仓/调仓）
- P&L 计算与归因
- 业绩指标（夏普、最大回撤、胜率、Sortino、卡玛比）

**性质**：纯代码，无 AI 调用。

### 2.4 投资助手层

**职责**：识别市场 regime、推荐策略权重、给出三层资产配置建议、维护胜率档案。

**包含**：
- Regime 分类器（基于规则的多维度状态判断）
- 资产配置规则引擎（现金/股票 + US/HK + 策略权重 三层）
- 策略推荐器（Regime → 各策略推荐权重的映射）
- 助手胜率档案（追踪助手历史建议的事后准确率）

**性质**：决策主体纯代码。简报文案由 AI 生成（**第二个允许的 AI 调用点**）。

### 2.5 应用层

**职责**：把核心系统的状态和建议暴露给用户。

**包含**：
- API 服务（FastAPI）：分公开路由和受保护路由两套
- 主动推送系统（触发规则 + 多渠道适配 + 降噪）
- 定时任务调度器（每日盘后跑分析，每月调仓等）

**性质**：纯代码。

### 2.6 前端层

**职责**：可视化和交互。

**包含**：
- 公开仪表盘（无登录可见）
- 策略管理页（登录后）
- 持仓和实施账户页（登录后）
- 历史策略库 + 回捞界面
- 助手胜率档案页

**性质**：纯前端代码（React + TypeScript）。

### 2.7 账户系统（独立模块，V0.7 引入）

**职责**：用户认证 + 券商凭证管理 + 实盘对接。

**包含**：
- 用户认证（JWT）
- 券商凭证加密存储
- 券商 API 适配器（IBKR / 富途等）
- 自动 / 手动推送两种执行模式

**性质**：纯代码，独立于其他层开发。

## 3. AI 边界原则

整个系统的核心架构铁律是 **AI 在边缘，代码在核心**。

### 3.1 仅有的两个 AI 调用点

**调用点 1：事件驱动策略中的新闻解析**

输入是非结构化的新闻文本，LLM 输出结构化的事件元数据（类型 + 影响方向 + 相关股票 + 置信度）。

- 输入：自然语言新闻
- 输出：符合 Pydantic Schema 的事件对象
- 为什么必须用 AI：新闻天然是文本，规则引擎无法泛化处理
- 约束：LLM 输出仅作为信号生成的**输入**，最终买卖决策由代码规则二次确认（量价确认、风控检查），LLM 输出不直接生成交易信号

**调用点 2：投资助手简报的叙事生成**

输入是结构化的市场状态数据，LLM 输出可读的中英文叙事文本。

- 输入：Regime 概率、推荐权重、当日数据
- 输出：自然语言简报段落
- 为什么必须用 AI：可读性和上下文连贯，模板化文本太死板
- 约束：LLM 只描述数据、不引入新判断；所有数字必须来源于结构化输入；失败时降级到 Jinja 模板

### 3.2 严格禁止的 AI 用法

下列场景**绝对不能**让 LLM 介入：

- LLM 直接判断市场 regime（必须用规则或 ML 分类器）
- LLM 直接选股（必须用因子打分公式）
- LLM 直接给出买卖价位（必须用 ATR、支撑压力、风报比公式）
- LLM 决定策略升级 / 降级（必须满足结构化升级条件）
- LLM 动态调整策略权重（权重由 Regime 映射表确定）

每次出现"让 LLM 直接判断会更聪明"的诱惑时，用同一句话回头检验：**这件事我能用历史数据回测吗？** 不能就是不该用 LLM。

### 3.3 LLM Gateway 中央化

所有 LLM 调用通过单一入口 `LLMGateway`，统一管理：

- 模型版本固定（不用 `claude-3-5-sonnet`，用 `claude-3-5-sonnet-20241022` 这种带日期的版本号）
- 温度归零（temperature=0.0）
- Pydantic Schema 强制结构化输出
- 调用日志和审计
- 降级路径（LLM 失败时回退到模板或规则）

业务代码绝不直接调用 LLM 库，必须经过 Gateway。

## 4. Champion-Challenger 框架

### 4.1 核心机制

策略池中同时运行多套策略，其中 1 个是冠军（Champion，绑定实施账户），其余是挑战者（Challenger，绑定影子账户）。所有策略看到同样的市场数据、产出各自的信号、独立维护持仓和 P&L。

业绩档案持续累积。当某个挑战者满足升级条件时，可以升级为冠军；原冠军降级回挑战者或归档。

### 4.2 活跃池组成

- **4 个内置永久策略**：价值反转、趋势动量、事件驱动、指数跟踪（GTAA）
- **0-2 个用户自定义策略**：基于内置策略的 4 维加权组合（V0.5 引入）

总活跃数量上限 6 个，用户可在设置中调整观察策略数量（默认 3，上限 6）。

### 4.3 影子账户初始资本规则

为保证 Champion-Challenger 业绩对比的公平性和可解释性，所有影子账户遵循以下资本约定：

**默认规则**：每个影子账户独立 100,000 美元（USD）起始资本。新策略加入活跃池或被回捞时，账户资本重置为该值。

**为什么是独立等额而非"总盘瓜分"**：让每个策略在"完整资金可用"的条件下表达自己的最佳行为，便于回测一致性。同一策略在 V0.2 单跑和 V0.7 接入实盘时的逻辑应当相同——后者只是把 Live 账户的真实资本注入。

**港股影子账户**：V0.2 阶段港股影子账户也以 100,000 USD 等值起始（按当日 USDHKD 汇率换算成港币入账）。**V0.2 不做跨市场资金统一池或动态汇率折算**——这层在 V0.6 之后才有意义。

**实施账户（Live）资本**：由用户在 V0.7 上传券商凭证时由真实账户余额决定，不再使用默认值。

### 4.4 升级机制

升级**不自动触发**，由用户拍板。系统在某个挑战者满足升级条件时主动提示：

- 至少 3 个月影子期
- 累计超额收益高于冠军
- 夏普比率高于冠军
- 最大回撤不显著恶化
- 投资助手提供宏观加权意见

用户可选择立即升级、继续观察、或忽略。所有升级动作进审计日志。

### 4.5 仓位过渡机制

升级时冠军的存量持仓采用**漂移退出**策略：旧持仓继续走旧策略的退出规则，新资金走新策略的入场规则。这避免了硬切仓造成的不必要交易摩擦。

### 4.6 公平比较原则

- 所有策略看到完全一样的市场数据和起始池
- 所有策略只能 forward-test，禁止历史数据"补课"
- 影子账户必须遵守和实施账户一样的资金 / 流动性约束 + 滑点 + 手续费

## 5. 策略生命周期

### 5.1 状态机

每个策略有三种状态：

- `ACTIVE`：在活跃池中，每日产出信号、绑定影子账户、累计业绩
- `ARCHIVED`：已归档，停止实时运行，保留代码和完整业绩档案
- `DELETED`：已删除，物理清除（除审计日志外）

### 5.2 转换动作

**归档（Archive）**：当用户想用新策略替换活跃池中某个用户定制策略时，原策略转为 ARCHIVED，最后业绩快照永久保存。

**回捞（Rescue）**：用户在历史库看到某个归档策略的过往业绩还不错，可以重新激活回到活跃池。**关键**：回捞 = 重新出生，影子账户重置，老业绩进归档展示但不接续到新的活跃运行。这避免"幸存者偏差作弊"。

**删除（Delete）**：对历史策略的永久清除，给用户清理混乱策略库的能力。删除是不可逆动作，UI 应有二次确认。

### 5.3 内置策略的不变性

4 套内置策略永远不能被归档或删除——它们是系统的"基础设施"，不是用户可处置的对象。UI 上对内置策略隐藏归档/删除按钮。

## 6. 内置策略的理论独立性

4 套内置策略代表 4 种**理论上独立**的 alpha 来源：

| 策略 | Alpha 来源 | 信号来源 | 时间维度 | 主导行情 |
|------|------|--------|--------|--------|
| 价值反转 | 估值差均值回归 | 估值因子 + 质量因子 | 月-年 | 震荡市、熊市底部 |
| 趋势动量 | 行为偏差延续 | 价格动量 + 盈利动量 | 周-月 | 单边牛市、风格延续期 |
| 事件驱动 | 信息处理速度 | 新闻 + 财报 + 监管 | 日-周 | 政策催化期、财报季 |
| 指数跟踪（GTAA） | 跨资产趋势分散 | 月线 200 日 SMA 过滤 | 月级 | 各种行情下的稳健基准 |

四者在金融学理上相互独立，海外学术研究中价值因子和动量因子月度相关系数长期在 -0.3 到 -0.5（著名的负相关组合）。这种独立性是 Champion-Challenger 框架价值的根基——**总有一个在状态**。

### 6.1 指数跟踪策略（GTAA）的特殊性

GTAA 是被动基准，不属于"主动选股"。它的存在保证了系统的诚实校准：用户的自定义主动策略必须实打实跑赢这个低成本被动方案，否则没有存在价值。

资产池：SPY / QQQ / 2800.HK / 3033.HK / TLT / GLD（约 6 个核心 ETF）。
规则：每月最后一个交易日检查，价格 > 10 月 SMA 持有，否则换现金/短债。在趋势中的 ETF 等权分配。

## 7. 用户定制策略

### 7.1 V0.5 实现形式

自定义策略 = 4 套内置策略的加权组合。每个 custom 策略由 4 个权重定义：

```
(w_value, w_momentum, w_event, w_index)
权重之和 = 1
```

每天 custom 策略对每只候选股票的综合得分 = 加权求和各内置策略对该股票的得分。信号生成基于综合得分排序。

举例：
- 保守组合：(0.2, 0.2, 0.1, 0.5) — 50% 跟指数，主动部分均衡
- 进攻组合：(0.3, 0.5, 0.2, 0.0) — 不要被动，主动偏动量
- 平衡组合：(0.33, 0.33, 0.34, 0.0) — 均衡主动

### 7.2 V1.x 远期：AI 代码生成

未来版本支持自然语言描述策略 → AI 调用代码生成能力 → 产出真正的可运行策略代码。但这要等基础架构非常稳定后才做，因为 AI 生成代码涉及沙箱执行、静态分析、安全过滤等复杂工程。短期不在路线图内。

## 8. 投资助手机制

### 8.1 三层输出

每天助手产出三层结构化建议：

**第一层：风险开关（Cash vs Equity）**
总仓位建议 0%-100%。当 regime 进入"风险释放期"或尾部风险信号触发时，建议降低总仓位。

**第二层：市场分配（US vs HK）**
两个市场的相对权重。基于 USD 强弱、AH 溢价、地缘风险等因子。

**第三层：策略分配（每个市场内的策略权重）**
基于当前 regime 给出每个市场内 4 套内置策略的推荐权重分布。

### 8.2 Regime 分类法

A 股驱动力分类法的美港股版本：

- **业绩驱动期**：盈利改善、估值修复，价值反转最优
- **资金驱动期**：流动性宽松、增量资金，趋势动量最优
- **政策/事件驱动期**：政策催化、题材轮动，事件驱动最优
- **风险释放期**：杀估值、波动放大，红利防御 + 降总仓位

### 8.3 主动推送

四类触发场景：

1. Regime 切换（最高优先级，年 6-10 次）
2. 关键宏观事件前置提醒（FOMC 前、CPI 前、财报季）
3. 冠军-挑战者动态（挑战者跑赢冠军、冠军大回撤）
4. 异常风险（VIX 突破、AH 溢价极端、地缘事件）

降噪机制：冷静期、等级分、摘要日报模式。原则是**宁可漏推也不滥推**。

### 8.4 胜率档案的公开透明

每一次助手判断都入库，事后 30/90 天验证。档案包括：

- 总分卡：累计判断数、平均胜率、Regime 准确率
- 建议明细列表：按时间倒序，每条带事后验证状态
- 专项切片：Regime 切换准确率、事件前判断准确率、不同 regime 下推荐准确率
- 信任度：每条新建议旁显示该助手在类似场景的历史准确率

档案默认全公开，未登录用户也能看。这是产品诚实性的核心展示。

## 9. 公开 vs 私有双模式

### 9.1 未登录用户能看到什么

- 4 套内置策略的当前状态、影子账户业绩、当日信号
- 当前 Regime 判断和理由
- 投资助手当日建议（三层输出）
- 助手胜率档案
- 历史策略库（任何用户的归档策略，但脱敏处理）

未登录用户拿到的是**完整的策略表现观察站 + 操作建议台**，可以自己拿着建议去券商手动操作。

### 9.2 登录后多出什么

- 个人实施账户的实时持仓和 P&L
- 创建自己的自定义策略
- 上传券商凭证、托管交易
- 选择自动 / 手动推送执行模式
- 个性化的策略库（自己创建的、归档的、回捞的）

### 9.3 API 路由分离

- 公开路由：`/api/public/*`，无需鉴权
- 受保护路由：`/api/private/*`，需要 JWT

前端两种视图（anonymous + authenticated）共享同一个 API 后端，权限边界严格区分。

## 10. 关键技术约束

### 10.1 稳定性的三个前提

- **确定性**：同样输入今天和一个月后跑出来一样
- **可回测性**：历史数据能完整复现行为
- **可追责性**：每个决策能追溯到具体输入、规则、版本号

### 10.2 LLM 使用的强制约束

1. 结构化输出强制（JSON Schema 校验）
2. 模型版本固定（不用 latest 别名）
3. 温度归零（temperature=0.0）
4. 永远有规则化降级路径
5. 完整审计日志

详见 `invariants.md`。

### 10.3 数据安全

- API key、券商凭证、密码、个人持仓数据**绝不进 git**
- 所有敏感配置通过 `.env` + 加密存储
- 公开仓库默认假设代码可见，敏感信息必须在代码外

### 10.4 货币与资金记账

跨市场（美股 + 港股）系统必须显式区分币种，否则跨市场组合 mark-to-market、P&L 归因、汇率风险都无法正确处理。

**强制约束**：

1. **`Currency` 枚举强制存在**：项目层面定义 `Currency(str, Enum) {USD, HKD}`（V1.x 港股 A 股扩展时再加 `CNY`）
2. **所有钱字段必须显式带币种**：`Stock`、`Account`、`Position`、`Trade` 必须有 `currency: Currency` 字段。`Signal` 不强制（信号跟随 Stock 所在市场推断），但 `buy_range`、`stop_loss`、`take_profit` 数字的币种必须能从关联的 `stock_code` + `market` 唯一确定
3. **市场—币种映射约定**：`Market.US → Currency.USD`，`Market.HK → Currency.HKD`。代码中通过 `currency_for_market(market)` 工具函数集中实现，禁止散布 `if market == US: return USD` 的判断逻辑
4. **跨币种聚合时显式标注汇率**：V0.2 阶段不做实时汇率折算（每个市场内部独立计算 P&L），V0.6 投资助手做"美港合计组合视角"时显式注入 `FXRate(date, USD/HKD)` 对象，绝不使用隐式默认值
5. **数据库列约定**：所有 `Numeric(20, 6)` 货币列必须配套一个 `currency CHAR(3) NOT NULL` 列（用 ISO 4217 三字母代码），不允许"靠上下文推断"

**为什么不在 Phase 0 就做**：Phase 0 启动时这层约束未列入 prompt 的硬性要求，Implementer 按规格实现没有错。**改为在 Phase 0.5 微版本里补丁性实现**——在 V0.1 启动前完成 `Currency` 枚举 + 4 个表的 `currency` 字段 + Alembic 增量迁移 + 测试。这样代价最低（仅 1 个 Claude Code 会话），且赶在所有数据消费方落地之前。

**反例（明确禁止）**：

- ❌ 用 `Decimal` 字段的注释"在 US 是 USD、在 HK 是 HKD"代替 `currency` 字段
- ❌ 让 `Account.cash` 缺省按账户绑定的策略推断币种
- ❌ 在跨币种相加时硬编码 1 USD = 7.8 HKD

> **修订说明**：本节为 v1.1 新增（v1.3 doc-only 同步进 repo）。Phase 0 实现侧未覆盖此约束，由 Phase 0.5 补丁性实现。详见 `docs/wbs.md` Phase 0.5 章节。

### 10.5 回测语义（v1.2 新增）

回测是策略孵化系统的"裁判"。回测引擎错一行可能让所有策略业绩失真，所以回测语义必须明确写死，不留"工程判断"空间。

**强制约束**：

1. **决策时点 = T 日收盘后**：策略在 T 日 step 时可访问截至 T 日 close 的全部数据（含 T 日 close 本身）。这是 `PointInTimeDataView.as_of` 的边界。

2. **执行时点 = T+1 开盘**：T 日生成的信号在 T+1 日用 `bar.open` 价格 + 滑点执行。这避免"已知执行价就决策"的隐式 look-ahead。
   - 推论：回测窗口的最后一个**可执行**交易日 = `end_date - 1`（end_date 当天的信号没有 T+1 数据可执行；引擎要么静默丢弃这一天的信号，要么在 BacktestResult 中标注 unexecuted）
   - **校准例外**：纯 buy-and-hold 类一次买入永不卖出的策略，可在测试中简化为 T 日 close 决策 + T 日 close 执行。这种简化必须在测试 docstring 显式声明，且仅限 `src/backtest/_calibration_strategies.py` 范围

3. **MTM 时点 = T 日收盘**：每个交易日**收盘后** mark-to-market，append 一个 `PerformanceSnapshot`。

4. **close vs adj_close 严格分离**：
   - 决策路径（entry / exit / SMA 突破 / 支撑压力）使用 `bar.close`：保证点位时点的真实性，避免拆股/分红污染信号
   - 收益归因（`daily_return` / `cumulative_return` / `nav` 序列输入）使用 `bar.adj_close`：剔除分红/拆股噪音，反映真实总回报
   - **mark-to-market（持仓估值）使用 `bar.close`**：当前真实头寸的现金等价价值
   - **任何混用必须在 PR 描述中说明理由**，由 reviewer 显式确认。这条由 `src/backtest/INVARIANTS.md #B1` 强制

5. **分红假设（V0.1）= 隐式再投资**：
   - V0.1 不模拟分红现金流；分红效应通过 `adj_close` 隐式表达
   - 因此 `nav` 序列（cash + shares × close）与基于 `adj_close` 计算的总回报序列**通常不精确相等**，差额 ≈ 分红再投资的等价收益
   - 这是已知的设计选择，不是 bug
   - V0.7 实盘对接券商时改为模拟真实分红现金流（届时新版本号文档化）

6. **拆股处理（V0.1）= 依赖 adj_close 自动调整**：
   - V0.1 不显式处理 corporate action 事件
   - yfinance 的 adj_close 已反向调整历史价格反映拆股
   - 持仓在拆股日穿越时，share count 维持名义不变，持仓估值用未调整的 close 与拆股后市价一致
   - V0.2 多策略并行时若发现误差再升级到显式 corporate action 处理

7. **执行成本模型（ExecutionCostModel）必须显式带 currency**：
   - 不同市场（US / HK / 未来 CN）费率结构完全不同，引擎不能写死币种假设
   - 引擎初始化必须接受 `ExecutionCostModel(slippage_bps, fee_per_share, min_fee, currency)`
   - 校验 `cost_model.currency == account.currency`，不一致时 raise
   - V0.1 仅实现 `US_DEFAULT_COST`，HK 留待 WP-1.2 / V0.2

8. **position_size_pct 基数 = 信号生成时 NAV**：
   - `Signal.position_size_pct` 的基数是当时的 NAV（cash + 全部持仓 MTM），不是当时的 cash
   - 这避免"连续买入快速耗尽现金"和"单日只能买 1/size_pct 个标的"的数学陷阱
   - 现金不足时按比例缩减 BUY 的 shares，记录 partial fill，不抛错

9. **同日多笔交易顺序**：
   - 先 SELL（释放现金）→ 再 BUY（消费现金）
   - BUY 之间按 `signal.confidence` 降序、再按 `stock_code` 升序排序（保证确定性，避免依赖 dict 遍历顺序）

**为什么 v1.2 才补**：Phase 0 启动 prompt 没列回测语义为硬性约束（它属于 V0.1 实施层细节）。在 WP-2.7 启动前补 v1.2 §10.5 是恰当时机：

- WP-2.7 是 V0.1 第一个使用这层约束的 WP
- 所有下游 WP（WP-2.3 趋势动量、WP-2.6 自定义、WP-3.x 助手）都依赖回测引擎产出的 PerformanceSnapshot 语义对齐
- 此时落地代价最低（无任何已实现回测代码需返工）

> **修订说明**：本节为 v1.2 新增。WP-2.7 启动前由 Architect 拍板，所有 V0.1 阶段触及回测/历史模拟的代码受此约束。详见 `docs/INVARIANTS.md #8` 与 `src/backtest/INVARIANTS.md`。

## 11. 与现有项目的关系

本项目可参考但**不直接 fork**已有的 [daily_stock_analysis (DSA)](https://github.com/ZhuLinsen/daily_stock_analysis)。

可以借鉴 DSA 的：
- 数据源适配层结构（YFinance / AkShare / Tushare 接入模式）
- LLM 渠道编排（LiteLLM 多模型 fallback）
- 推送通道适配（Telegram / 企业微信 / 飞书 / Email）
- Pipeline Agent + DB-first 缓存模式

不直接复用的原因：
- DSA 的核心定位是"日报推送"，不是策略孵化
- DSA 没有 Champion-Challenger、影子账户、回测、用户定制等核心机制
- 从零构建反而更清爽，避免在它的架构里硬塞新概念

DSA 的代码值得通读一遍学习其工程实践，但实现层面我们建立自己的代码库。
